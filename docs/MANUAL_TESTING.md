# Employee Portal — Manual Testing Checklist

Use this when validating a build by hand (e.g. before a release, or after a non-trivial change). Each section is a self-contained scenario you can run from a freshly seeded database.

Prerequisites:

```bash
npm install
cp .env.example .env
npm run db:push
npm run db:seed
npm run dev
```

- Frontend: <http://localhost:5173>
- Backend: <http://localhost:4000>

Use private/incognito windows when switching between roles so cookies don't cross over.

---

## 0. Pre-flight checks (~30 s)

- [ ] Open <http://localhost:4000/api/health> — should return `{"ok":true}`.
- [ ] Open <http://localhost:5173/login> — login form renders with email + password fields and the seeded credentials hint.
- [ ] Open <http://localhost:5173/> while logged out — you are redirected to `/login`.

## 8. Role enforcement & API surface

Use the dev-tools network tab or `curl`. (The Playwright suite covers the API probe automatically; this is the manual equivalent.)

- [ ] As HR, copy your access token from `localStorage` / Authorization header.
- [ ] `curl -H "Authorization: Bearer $HR_TOKEN" http://localhost:4000/api/employees` → 200 + JSON array.
- [ ] Log out, log in as an employee, repeat with that token → **403 Forbidden**.
- [ ] Without any token → **401 Unauthorized**.
- [ ] As employee, hit `GET /api/payslips` → only that employee's payslips. As HR with `?employeeId=<id>` → that one employee's payslips.

---

## 9. Session & token handling

- [ ] Log in. Open dev-tools → Application → Cookies → confirm `refresh_token` (httpOnly) on the domain, path `/api/auth`.
- [ ] In dev-tools, delete `localStorage` and refresh the page. You stay logged in (silent refresh in `AuthContext` re-issues the access token from the refresh cookie).
- [ ] Click **Log out** → the cookie is cleared (`document.cookie` no longer lists it).
- [ ] After logout, refreshing the page leaves you on `/login`.

---

## 10. Negative / edge cases

- [ ] Stop the backend (Ctrl-C the `backend` row in the terminal). In the running UI, click any data link → an error is shown (TanStack Query) but the page doesn't crash. Restart backend; navigating again recovers.
- [ ] Hit a non-existent route like `/does-not-exist` while logged in → falls through to the layout's empty Outlet (you'll see the sidebar with no content). Logged out → redirected to `/login`. *(Optional improvement: a dedicated 404 page.)*
- [ ] Upload a payslip just under the 5 MB limit → succeeds. Just over (e.g. 6 MB) → 413/400 (Multer rejects with payload-too-large).
- [ ] Concurrent assignment race: if you're feeling adventurous, fire two `POST /api/assets/:id/assign` requests in the same second. Expected: one succeeds, the other returns 400 "Asset already assigned". *(Note: this check is not strictly atomic — see `backend/src/routes/assets.routes.ts`.)*

## 11. Cleanup / reset to a known state

```bash
# wipe + reseed
rm -f backend/prisma/dev.db backend/prisma/dev.db-journal
rm -rf backend/uploads/payslips/*
npm run db:push
npm run db:seed
```

After this, the credentials and asset state listed in this document are again accurate.
